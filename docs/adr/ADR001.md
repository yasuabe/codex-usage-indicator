# ADR001: 利用量情報源として rollout snapshot を採用する

## ステータス

採用

## 文脈

`codex-usage-indicator` は、Codex の利用量を GNOME AppIndicator で表示するための
個人用ツールである。

当初は Codex の `/status` に近いライブ値を取得したかったが、少なくとも調査時点では
公開 API は見当たらず、`/status` の表示内容を安定して外部取得する方法も確立できなかった。

一方で、`~/.codex/sessions/**/rollout-*.jsonl` には `token_count` イベントが永続化され、
その一部には `rate_limits` が含まれることを確認した。

## 確認できた事実

- `/status` だけでは、少なくとも直後には `rollout` と `state_5.sqlite.threads` への
  `rate_limits` 永続化を確認できなかった
- interactive セッションで通常入力を 1 回送ると、`rollout` と
  `state_5.sqlite.threads` が生成され、`token_count.rate_limits` が入った
- `gpt-5.1-codex-mini` でも通常ターン後に同様の永続化が起き、
  `rate_limits.limit_id` は `codex` のままだった
- 一部の非対話経路では `token_count` は出ても `rate_limits = null` だった
- セッション終了時の追加 flush は、少なくとも今回の観測では確認できなかった

## 決定

利用量の情報源として、`~/.codex/sessions/**/rollout-*.jsonl` に保存された
`token_count.rate_limits` を採用する。

表示値は次のように定義する。

- このマシン上の全 rollout を対象にする
- `payload.type == "token_count"` かつ `payload.rate_limits != null` の候補を集める
- その中で `timestamp` が最も新しい候補を採用する
- 同時刻候補が複数ある場合は、より逼迫している候補を採用する
  - `primary.used_percent` が大きい方を優先
  - さらに同じなら `secondary.used_percent` が大きい方を優先

したがって、このアプリケーションが表示する利用量は
「全セッションの中で最後に永続化された non-null `rate_limits` snapshot」であり、
`/status` のライブ現在値ではない。

## 採用しなかった案

### `/status` のライブ表示をそのまま取得する

PTY 経由で `/status` を実行し、画面出力から必要行を抜く案を試した。
しかし、取得できたのはフッター断片までで、`5h limit` / `Weekly limit` の本文を
安定取得できなかったため不採用とした。

### `tmux` を補助的に使って画面バッファを取得する

補助実験としては有効だったが、`tmux` が入っていない環境では使えない。
「新規環境でも動くこと」と相性が悪いため、最終実装の依存にはしない。

### `codex exec` や起動時 prompt 引数で 1 ターン発生させる

非対話経路で `token_count` を出すこと自体はできたが、観測できたケースでは
`rate_limits = null` だったため、安定した取得経路として採用しなかった。

## 帰結

- 表示値は実用上有用だが、ライブ現在値ではない
- 複数端末で Codex を使っていても、このマシンに永続化された最新 snapshot は拾える
- 進行中ターンの消費や、まだ永続化されていない利用量は反映されない
- 利用者が鮮度を判断できるよう、更新時刻と snapshot 時刻を併記する
